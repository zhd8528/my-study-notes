## 1. TCP 三次握手

![三次握手](https://zhd-blog.oss-cn-beijing.aliyuncs.com/4c205325a47b4c159331e2b6bb8f06f1.png)

**为什么不是两次？**

> **1. 防止历史连接请求**：
>
> 客户端发起的过期连接请求（如网络延迟）如果服务器只回复一次就建立连接，会浪费资源
>
> **2. 确认双方收发能力**：
>
> - 第一次：客户端发送能力 OK
> - 第二次：服务端接收 + 发送能力 OK
> - 第三次：客户端接收能力 OK
>
> 两次握手只能保证服务端的收发能力，无法确认客户端的接收能力

## 2. TCP 四次挥手

![四次挥手](https://zhd-blog.oss-cn-beijing.aliyuncs.com/ae26673e8ef54009a90e3296186cbc15.png)

**四次握手状态流转**：

> 主动关闭方（Client）：
>
> `ESTABLISHED`->`FIN_WAIT_1`->`FIN_WAIT_2`->`TIME_WAIT`->`CLOSED`
>
> 被动关闭方（Server）：
>
> `ESTABLISHED`->`CLOSE_WAIT`->`LAST_ACK`->`CLOSED`

## 4. GET VS POST

| 对比项   | GET                    | POST                          |
| -------- | ---------------------- | ----------------------------- |
| 参数位置 | URL 查询字符串         | Request Body                  |
| 长度限制 | 有（浏览器限制 2k~8k） | 无（理论上）                  |
| 安全性   | 低（参数暴露在 URL）   | 较高（但在 HTTPS 中也安全）   |
| 幂等性   | ✅️（多次请求结果相同）  | ❌️（多次请求可能创建多个资源） |
| 缓存     | ✅️（默认可缓存）        | ❌️（默认不缓存）               |
| 历史记录 | ✅️（保留在浏览器历史）  | ❌️（不保留）                   |
| 使用场景 | 查询、获取数据         | 提交、创建、修改数据          |

## 追问

### 1. TCP 四次挥手细节

**为什么需要四次挥手？**

> TCP 是全双工的，双方都要关闭各自的连接
>
> 主动方发送 FIN 表示不再发送数据，但还能接收；
>
> 被动方先回复 ACK，等自己发完数据后再发 FIN

**TIME_WAIT 为什么是 2MSL？**

> MSL 是报文最大生存时间
>
> 2MSL 确保被动方收到最后的 ACK，并且让旧连接的所有报文都从网络中消失，防止影响旧连接

**CLOSE_WAIT 和 TIME_WAIT 哪个是主动方的状态？**

> TIME_WAIT 是主动发送方（先发 FIN）；
>
> CLOSE_WAIT 是被动关闭方（收到 FIN 后回复 ACK，等自己发 FIN）

**大量 TIME_WAIT 有什么问题？怎么解决？**

> 占用端口资源，可能导致无端口可用
>
> 解决方案：
>
> | 方案         | 命令/配置                                   | 说明                                  |
> | ------------ | ------------------------------------------- | ------------------------------------- |
> | 开启重用     | `net.ipv4.tcp_tw_reuse = 1`                 | 允许将 TIME_WAIT 的 socket 用于新连接 |
> | 调小超时     | `net.ipv4.tcp_fin_timeout = 30`             | 默认 60 秒，可调小                    |
> | 调整端口范围 | `net.ipv4.ip_local_port_range = 1024 65000` | 扩大可用端口                          |
> | 开启快速回收 | `net.ipv4.tcp_tw_recycle = 1`               | ❗️ 不推荐（NAT 环境下有问题）          |

### 2. TCP 与 UDP 的区别

| 对比项 | TCP              | UDP                 |
| ------ | ---------------- | ------------------- |
| 连接   | 面向连接         | 无连接              |
| 可靠性 | 可靠（确认重传） | 不可靠（可能丢包）  |
| 速度   | 慢               | 快                  |
| 应用   | HTTP、FTP、SSH   | DNS、视频通话、游戏 |

**为什么视频用 UDP？**

> 实时性要求高于可靠性，丢一帧无所谓，但延迟不能高

### 3. TCP 如何保证可靠传输

**机制**：

> **1. 序列号 + 确认应答**：
>
> 每个字节有编号，ACK 告知已收到
>
> **2. 超时重传**：
>
> 发送后若超时未收到 ACK，则重传
>
> **3. 流量控制（滑动窗口）**：
>
> 接收方告诉发送方自己的缓冲区大小，防止发太快
>
> **4. 拥塞控制**（4 个阶段）：
>
> | 阶段 | 算法     | 行为                                         |
> | ---- | -------- | -------------------------------------------- |
> | 1    | 慢启动   | 指数增长窗口（1 -> 2 -> 4...）直到慢启动阈值 |
> | 2    | 拥塞避免 | 线性增长窗口（每次 + 1）                     |
> | 3    | 快重传   | 收到 3 个重复 ACK 立即重传，不等超时         |
> | 4    | 快恢复   | 拥塞窗口减半，进入拥塞避免                   |

### 5. GET VS POST 补充

**什么是幂等？**

> 一次或多次请求，资源状态相同
>
> GET是安全的也是幂等的；
>
> POST 不安全也不幂等（多次提交会创建多个资源）

**PUT 和 DELETE 的幂等性？**

> PUT 幂等（更新相同数据多次结果一样）；
>
> DELETE 幂等（删除多次结果一样）

### 6. HTTPS 握手过程

**HTTPS 如何保证安全？**

> 混合加密：
>
> 非对称加密传输/对称密钥，对称加密传输数据

**简要握手过程？**

1. 客户端发支持的加密套件
2. 服务端发证书（公钥）
3. 客户端验证证书，生成随机密钥（用公钥加密）发送
4. 服务端解密得到对称密钥
5. 后续通信用对称加密

**证书的作用？**

> 防止中间人攻击，证明服务端身份（CA 签发）

**TLS 1.2 握手（4 次 RTT）**：

1.  Client Hello
2. Server Hello + Certificate
3. Client Key Exchange + Change Cipher Spec
4. Change Cipher Spec + Finished

**TLS 1.3 握手（1 次 RTT，支持 0-RTT）**：

- 减少了往返次数
- 废弃了不安全的加密套件
- 支持 0-RTT 恢复会话

### 7. HTTP 版本演进

**HTTP/1.0**：

> 短连接，每次请求重新建立 TCP

**HTTP/1.1**：

> 持久连接（keep-alive）、管道化（但仍有队头阻塞）

**HTTP/2**：

> 二进制分帧、多路复用、服务器推送、头部压缩

**HTTP/3**：

> 基于 UDP 的 QUIC 协议，解决队头阻塞，连接迁移

**HTTP 版本演进总结**：

| 版本 | 特点                           | 缺点                               |
| ---- | ------------------------------ | ---------------------------------- |
| 1.0  | 短连接                         | 每次请求都要重新建立 TCP           |
| 1.1  | 持久连接、管道化               | 队头阻塞（一个请求慢了后端的都等） |
| 2.0  | 多路复用、头部压缩、服务器推送 | TCP 队头阻塞（TCP 丢包影响所有流） |
| 3.0  | QUIC 协议（UDP 实现可靠传输）  | 普及率不高，对网络设备要求高       |

### 8. Cookie 与 Session

**Cookie 和 Session 区别？**

> Cookie 存客户端，Session 存 服务端
>
> Session 通过 Cookie 中 的 sessionId 关联

**禁用 Cookie 怎么办？**

> URL 重写（把 sessionId 拼在 URL 里）

**Cookie/Session VS JWT**：

| 对比项   | Cookie + Session          | JWT                    |
| -------- | ------------------------- | ---------------------- |
| 存储位置 | 服务端存储 Session        | 客户端存储 Token       |
| 扩展性   | 需要共享 Session（Redis） | 无状态、天然支持分布式 |
| 安全性   | 需防范 CSRF               | 需防范 XSS             |
| 注销     | 服务端删除即可            | 需黑名单或短过期时间   |
| 适用场景 | 传统 Web 应用             | 前后端分离、移动端     |

### 9. 输入 URL 到页面显示（核心步骤）

**1. DNS 解析**：域名 -> IP 地址

**2. TCP 连接**：三次握手

**3. TLS 握手**（HTTPS）：建立安全连接

**4. 发送 HTTP 请求**：请求资源

**5. 服务器处理**：返回响应（HTML / CSS / JS）

**6. 浏览器解析渲染**：

- 解析 HTML 构建 DOM 树
- 解析 CSS 构建 CSSOM 树
- 合并为渲染树
- 布局（Layout）-> 绘制（Paint）-> 合成（Composite）

**7. TCP 断开**：四次挥手

## 可选补充

### 1. TCP 粘包/拆包

> 流式协议，消息边界问题，用定长、分隔符、长度字段解决

**产生原因**：

> TCP 是流式协议，没有消息边界

**解决方案**：

| 方案       | 示例                                    |
| ---------- | --------------------------------------- |
| 定长消息   | 每个消息固定长度 1024 字节              |
| 特殊分隔符 | 用`\r\n`或`\0`分隔                      |
| 长度字段   | 消息头带长度（如 4 字节表示 body 长度） |

**Netty 中的实现**：

- `FixedLengthFrameDecoder`（定长）
- `DelimiterBasedFrameDecoder`（分隔符）
- `LengthFieldBasedFrameDecoder`（长度字段）

### 2. DNS 解析过程

**递归查询**（客户端 + 本地 DNS）：

> 客户端只问一次，DNS 服务器负责查到底

**迭代查询**（本地 DNS -> 根域名服务器 -> 顶级域 -> 权威域名服务器）：

> DNS 服务器返回下一级地址，让客户端自己继续查

**整体流程**：

1. 浏览器缓存 -> 操作系统 hosts -> 本地 DNS 缓存
2. 本地 DNS 服务器递归/迭代查询
3. 返回 IP 地址
