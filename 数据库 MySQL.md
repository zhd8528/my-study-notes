## 1. 事务 ACID

**原子性**：

> 全部成功或全部失败（undo log）

**一致性**：

> 事务前后数据一致，其他三个特性共同保证

**隔离性**：

> 事务互不干扰（锁 + MVCC）

**持久性**：

> 提交后永久保存（redo log）

---

## 2. 隔离级别

| 级别                          | 脏读（是否允许） | 不可重复读（是否允许） |
| ----------------------------- | ---------------- | ---------------------- |
| Read Uncommitted              | ✅️                | ✅️                      |
| Read Committed                | ❌️                | ✅️                      |
| Repeatable Read（MySQL 默认） | ❌️                | ❌️                      |
| Serializable                  | ❌️                | ❌️                      |

---

## 3. MVCC 原理

> 通过隐藏列（`DB_TRX_ID`、`DB_ROLL_PTR`） + ReadView 实现

**不同隔离级别的 ReadView 创建时机**：

> 可重复读：
>
> - ReadView 在事务开始时创建（只创建一次）
>
> 读已提交：
>
> - ReadView 在每次查询时创建

---

## 4. 索引

**聚簇索引**：

> 数据和索引一起存，主键自带，叶子节点完整行数据

**二级索引**：

> 存主键值，需回表

**最左前缀**：

> 联合索引`(a，b，c)`能走 `a`、`a.b`、`a,b.c`，不能走`b,c`

----

## 5. 索引失效场景

- LIKE '%XXX' （以 % 开头）
- 违反最左前缀
- 索引列参与运算、使用函数
- OR 连接（除非所有条件都有索引）
- 隐式类型转换（`varchar`列预数字比较）
- `!=`或`<>`
- `IS NULL` / `IS NOT NULL`（视数据分布而定）

---

## 6. Explain 重点

**type**：

> 性能从好到差：
>
> 1. system
> 2. const
> 3. eq_ref
> 4. ref
> 5. range
> 6. index
> 7. ALL

**type 含义**：

| type   | 含义                        | 典型场景              |
| ------ | --------------------------- | --------------------- |
| system | 只有一行记录的系统表        | 极少                  |
| const  | 主键/唯一索引等值查询       | `WHERE id = 1`        |
| eq_ref | 联表查询，使用主键/唯一索引 | JOIN ON 主键          |
| ref    | 非唯一索引等值查询          | `WHERE name = '张三'` |
| range  | 索引范围扫描                | `WHERE id > 10`       |
| index  | 全索引扫描                  | 遍历整个索引树        |
| ALL    | 全表扫描                    | 无索引或索引失效      |

**Extra**：

> 出现 Using filesort / Using temporary 说明需优化

## 追问

### 1. 事务隔离级别补充

| 级别                          | 脏读（是否允许） | 不可重复读（是否允许） |幻读（是否允许）|
| ----------------------------- | ---------------- | ---------------------- | ---------------------- |
| Read Uncommitted              | ✅️                | ✅️                      |✅️|
| Read Committed                | ❌️                | ✅️                      |✅️|
| Repeatable Read（MySQL 默认） | ❌️                | ❌️                      |❌️（间隙锁防幻读）|
| Serializable                  | ❌️                | ❌️                      |❌️|

**MySQL 的 RR 级别如何解决幻读？**

> 使用间隙锁（Gap Lock）
>
> 例如 SELECT * FROM t WHERE id > 10 FOR UPDATE，不仅锁住现有行，还锁住（10，+∞）这个间隙，防止其他事务插入新行

---

### 2. MVCC 细节补充

**ReadView 的 3 个关键属性**：

| 属性         | 作用                               |
| ------------ | ---------------------------------- |
| trx_ids      | 当前活跃（未提交）的事务 ID 列表   |
| low_limit_id | 下一个待分配的事务 ID （低水位）   |
| up_limit_id  | 活跃列表中最小的事务 ID （高水位） |

**可见性判断规则**（以 RR 为例，给定`DB_TRX_ID`）：

> 如果 `DB_TRX_ID < up_limit_id`：
>
> - **可见**（事务已提交）
>
> 如果 `DB_TRX_ID >= low_limit_id`：
>
> - **不可见**（事务在未来）
>
> 如果 `up_limit_id <= DB_TRX_ID < low_limit_id`（中间值）：
>
> - 在 `trx_ids` 中
>   - **不可见**（未提交）
> - 不在 `trx_ids` 中
>   - **可见**（已提交）

### 3. 索引失效场景补充

**除了**：[索引失效场景](##5. 索引失效场景)

**还有**：

> **隐式类型转换**：
>
> - 例如：WHERE phone = 13800000000（phone 是 vachar），优化器会全表扫描
>
> **!= 或 <>**：
>
> - 例如：WHERE age <> 20，非等值通常走不了索引
>
> **IS NULL / IS NOT NULL**：
>
> - 有时有效，有时失效，取决于列的可空性和数据分布

### 4. Explain 的 Extra 字段具体含义

| 字段            | 含义                                         | 严重程度 |
| --------------- | -------------------------------------------- | -------- |
| Using filesort  | MySQL 需额外排序（无法用索引排序）           | 必须优化 |
| Using temporary | 用临时表（常见于 group by 无索引、distinct） | 严重     |
| Using index     | 覆盖索引（不回表）                           | 好       |
| Using where     | 用 where 过滤，但可能没走索引                | 需检查   |

### 5. SQL优化实战场景

**分页大偏移量 OFFSET 10000**：

> 优化手段：
>
> - 延迟关联：
>   - SELECT * FROM t JOIN (SELECT id FROM t ORDER BY id LIMIT 10000, 10) tmp ON t.id = tmp.id

**COUNT(*) 慢**：

> 优化手段：
>
> - MyISAM 有专门计数
> - InnoDB 用二级索引计数（如 `COUNT(1)` 走最小索引）
>
> 优化细节：
>
> - `COUNT(*)` 和 `COUNT(1)` 在 InnoDB 中性能相同，都会选最小的二级索引
> - `COUNT(列)` 会检查该列是否为 NULL，通常更慢
> - 想要精确行数且要求性能，考虑用额外表维护计数或使用 MyISAM （但不推荐）

**IN 子查询**：

> 优化手段：
>
> 尽量改写成 EXISTS 或 JOIN，因为 MYSQL 旧版对 IN 子查询优化差

### 6. 主从复制与读写分离

**主从复制原理**：

1. 主库记录 binlog
2. 从库 I/O 线程拉取并写入 relaylog
3. 从库 SQL 线程重放

**binlog 格式**：

> STATEMENT（记录 SQL）、ROW（记录行变化，安全但占空间）、MIXED

**读写分离可能的问题**：

> 主从延迟导致读不到刚写入的数据
>
> 解决：
>
> - 强制读主、延迟容忍、半同步复制
>
> 半同步复制：
>
> - 主库等待至少一个从库确认收到 binlog 后才返回提交成功
> - 参数 `rpl_semi_sync_master_timeout` 控制超时（超时后降级为异步）
> - 平衡数据安全与性能

---

### 7. 分库分表基础

**垂直拆分 VS 水平拆分**：

> 垂直：
>
> - 按列拆分（如热数据表、冷数据表）
>
> 水平：
>
> - 按行拆分（如按 user_id hash）

**分片键选择原则**：

> 查询频率高、分布均匀、不参与更新

**全局唯一 ID 方案**：

> 雪花算法、Leaf（美团）、号段模式

**雪花算法（64 bit）**：

| 位     | 用途                         |
| ------ | ---------------------------- |
| 1 bit  | 符号位（0）                  |
| 41 bit | 时间戳（毫秒级、可用 69 年） |
| 10 bit | 机器 ID（最多 1024 节点）    |
| 12 bit | 序列号（每毫秒 4096 个 ID）  |

### 2. 自适应哈希索引

> InnoDB 自动建立，加速热点索引页的等值查询

### 3. Online DDL

> MYSQL 5.6 + 支持，ALTER TABLE ... ALGORITHM = INPLACE 允许并发读写

**示例**：

```sql
-- 添加列，允许并发读写
ALTER TABLE t ADD COLUMN c INT, ALGORITHM=INPLACE, LOCK=NONE;

-- 修改列类型，需要表锁
ALTER TABLE t MODIFY COLUMN c VARCHAR(100), ALGORITHM=COPY, LOCK=SHARED;
```

### 4. InnoDB VS MyISAM 对比

| 特性     | InnoDB        | MyISAM    |
| -------- | ------------- | --------- |
| 事务     | ✅️             | ❌️         |
| 行级锁   | ✅️             | ❌️（表锁） |
| 外键     | ✅️             | ❌️         |
| 聚簇索引 | ✅️             | ❌️         |
| 全文索引 | ✅️（5.6 +）    | ✅️         |
| 崩溃恢复 | ✅️（redo log） | ❌️         |

---

### 5. Redo Log VS Binlog

| 特性     | Redo Log           | Binlog                   |
| -------- | ------------------ | ------------------------ |
| 所属层   | InnoDB 引擎层      | MYSQL Server 层          |
| 内容     | 物理日志（页修改） | 逻辑日志（SQL 或行变化） |
| 用途     | 崩溃恢复（持久性） | 主从复制、数据恢复       |
| 写入时机 | 事务执行过程中     | 事务提交时               |
