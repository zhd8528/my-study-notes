## 1. 内存区域

| 区域       | 存放内容               | 线程共享 |溢出类型|
| ---------- | ---------------------- | -------- |-------|
| 堆         | 对象实例、数组         | ✅️        |OutOfMemoryError|
| 方法区     | 类信息、静态变量、常量 | ✅️        |OutOfMemoryError（元空间）|
| 虚拟机栈   | 局部变量、方法调用     | ❌️        |StackOverflowError/OOM|
| 程序计数器 | 当前执行字节码行号     | ❌️        |无|

## 2. GC 判断对象存活

- **可达性分析**：从 GC Roots 出发，不可达即回收
- **GC Roots**：栈帧中局部变量、静态变量、JNI 引用等

## 3. 常用 GC 算法

| 算法      | 原理                                 | 优点   | 缺点                | 适用区域                 |
| --------- | ------------------------------------ | ------ | ------------------- | ------------------------ |
| 标记-清除 | 标记存活对象，清除未标记的           | 简单   | 产生内存碎片        | 老年代（配合 CMS）       |
| 复制      | 将存活对象复制到另一半空间           | 无碎片 | 内存利用率低（50%） | 新生代                   |
| 标记-整理 | 标记存活对象，向一端移动，清理边界外 | 无碎片 | 移动对象成本高      | 老年代（G1、Shenandoah） |



## 4.  类加载机制

**类加载过程**

> 加载、验证、准备、解析、初始化

| 阶段   | 做了什么                     |
| ------ | ---------------------------- |
| 加载   | 查找并加载类的二进制数据     |
| 验证   | 确保类文件格式正确、语义合法 |
| 准备   | 为静态变量分配内存并赋零值   |
| 解析   | 将符合引用转换为直接引用     |
| 初始化 | 执行静态代码块和静态变量赋值 |



**双亲委派**

- 类加载请求先委派父类，避免重复加载 + 保证核心内安全

## 追问

### 1. 内存区域补充

==**堆内存如何分代？各代存放什么？**==

- **年轻代**：Eden、S0、S1，存放新创建的对象

- **老年代**：存放存活多次的对象（年龄阈值默认 15）或大对象

==**方法区 VS 元空间 VS 永久代**==

> JDK8 以前方法区实现叫永久代（Perm Gen），JDK8 开始用元空间（Metaspace），使用本地内存而非堆内存，避免OOM

**程序计数器为什么是唯一没有 OOM 的区域？**

> 每个线程私有，只存储下一条指令地址，内存固定，不会增长

**栈溢出（StackOverflowError）和堆溢出（OutOfMemoryError:Java heap space）分别什么场景？**

- **栈溢出**：递归调用太深
- **堆溢出**：大量对象无法回收
  - 内存泄露或数据量过大

### 2. GC 判断存活补充

**GC Roots 具体有哪些？**

- **栈引用**：虚拟机栈（栈帧中的局部变量表）引用的对象

- **静态引用**：静态属性引用的对象

- **常量引用**：常量池引用的对象

- **JNI 引用**：Native 方法引用的对象

- **活跃线程**：所有活跃的线程对象

**可达性分析中，被标记为不可达的对象一定会被回收吗？**

- 不一定
- 对象可以自救，在`finalize() 方法中重新与 GC Roots 关联`
- 不推荐使用`finalize()`，因为
  - 执行时机不确定
  - 只会执行一次
  - 性能差，已被标记为 deprecated（Java 9+）


==**4  种引用类型在 GC 时的行为**==

- **强引用**：不回收
- **软引用**：内存不足时回收
- **弱引用**：下次 GC 回收
- **虚引用**：任何时候都可能回收，主要用于追踪对象回收

### 3. GC 算法补充

==**标记-整理算法**==

> 标记存活对象，然后向一端移动，最后清理边界以外的内存。
>
> 解决标记-清除产生的碎片问题，老年代常用。

**分代收集理论**

- **新生代**：大量对象朝生暮死，适合复制算法
- **老年代**：存活率高，适合标记-清除或标记-整理

**Minor GC VS Major GC VS Full GC**

- **Minor GC**：回收新生代（频繁）
- **Major GC**：回收老年代（少见）
- **Full GC**：回收整个堆 + 方法区（代价大）

### 4. 垃圾回收器

==**G1 有什么特点？**==

- 将堆划分为多个大小相等的 Region（1~32MB）
- 每个 Region 可以是 Eden、Survivor、Old 或 Humongous（大对象）
- 可预测停顿时间模型，优先回收垃圾最多的 Region（混合回收）
- JDK9+ 默认

**CMS 为什么废弃?**

> 并发标记阶段产生浮动垃圾，且内存碎片化严重，Full GC 时退化为 Serial 收集器，停顿长

### 5. 类加载机制补充

**3 个类加载器各加载什么？**

- **引导类加载器（Bootstrap）**：加载`rt.jar`
- **扩展类加载器（Extension）**：加载`lib/ext`
- **应用类加载器（App）**：加载 `classpath`

==**双亲委派模型的好处？**==

- 避免类重复加载，防止核心类被篡改
  - 比如自定义`java.lang.String`不会生效

==**如何破坏双亲委派？举例**==

- 实现`ClassLoader`并重写`loadClass()`而非`findClass()`
- 如 Tomcat 的`WebappClassLoader`优先加载 Web 应用自己的类

==**准备阶段和初始化阶段的区别？**==

- 准备阶段为静态变量分配内存并赋零值
- 初始化阶段执行静态代码块和静态变量赋值

### ==6. OOM 排查==

**产生堆转储文件（heap dump）后怎么分析？**

> 使用 jmap 生成 dump，用 MAT（Memory Analyzer Tool）或 VisualVM 分析，查看大对象和引用链

- 生成 heap dump

```bash
# 方式1：jmap
jmap -dump:live,format=b,file=heap.hprof <pid>

# 方式2：JVM 参数自动导出
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump
```
- 分析工具
  - **MAT**：查找大对象、支配树、泄漏疑点
  - **VisualVM**：实时监控、离线分析
  - JProfiler / Arthas（生产环境）


**常见 OOM 类型**

1. 堆 OOM（对象太多）
2. 栈 OOM（递归或线程过多）
3. 元空间 OOM（动态生成类过多，如大量代理类）
4. 直接内存 OOM（DirectByteBuffer未释放）

### 7. JVM 调优常用参数
| 参数 | 作用 |
|:-----|:-----|
| `-Xms` / `-Xmx` | 堆初始/最大大小 |
| `-Xmn` | 新生代大小 |
| `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize` | 元空间初始/最大 |
| `-XX:SurvivorRatio` | Eden/Survivor 比例（默认 8）|
| `-XX:MaxTenuringThreshold` | 晋升老年代年龄阈值（默认 15）|
| `-XX:+UseG1GC` | 使用 G1 收集器 |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时自动 dump |
| `-XX:OnOutOfMemoryError` | OOM 时执行脚本 |
