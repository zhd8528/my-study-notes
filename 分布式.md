## 1. CAP 理论

| C        | Consistency（一致性）                              | 所有节点在同一时间看到相同的数据                             |
| -------- | -------------------------------------------------- | ------------------------------------------------------------ |
| A        | Availability（可用性）                             | 每个请求都能收到非错误响应（不保证数据最新）                 |
| P        | Partition tolerance（分区容错性）                  | 系统在网络分区（节点间丢包 / 断连）时仍能工作                |


**核心理论**：

> 分布式系统 P 必须满足（网络分区总会发生），只能在 C 和 A 之间权衡

**1. CP 系统**：

> 保证一致性和分区容错（如 Zookeeper、Etcd、HBase），牺牲部分可用性，保证一致性（可能出现超时或拒绝）

**2. AP 系统**：

> 保证可用性和分区容错（如 Eureka、Cassanda、Riak），牺牲强一致性，保证可用性（最终一致）

**3. CA 系统**：

> 理论上不存在（因为不满足 P 就不是分布式系统，就是单机）
>
> 单机 MYSQL，不存在于分布式系统

---

## 2. BASE 理论（CAP 的延伸）

| BA   | Basically Available（基本可用）   | 允许部分功能降级，不要求随时全量可用 |
| ---- | --------------------------------- | ------------------------------------ |
| S    | Soft state（软状态）              | 允许中间状态（如数据副本不一致）     |
| E    | Eventually consistent（最终一致） | 经过一段时间后，数据最终达到一致     |

## 3. 分布式锁（如果项目用过 Redis 分布式锁）

**如何用 Redis 实现分布式锁？**

> `SET key value NX EX seconds`（原子设置锁 + 过期时间）
>
> 释放时用 Lua 脚本：
>
> 先 GET 比较 value （防止误删别人锁），再 DEL

**释放（Lua 脚本保证原子性）**：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**有什么问题？**

> 锁过期时间难设置：
>
> Redission 看门狗自动续期解决
>
> Redis 主从异步复制导致锁丢失：
>
> RedLock （多数节点成功）

**RedLock 算法思路？**

> 向多个独立 Redis 节点请求锁，超过半数成功才认为获得锁
>
> 不常用，因为复杂度高，Redis 官方不推荐

## 4. 分布式事务（如果简历上有）

**2PC（两阶段提交）**：

> 阶段一（准备）：
>
> 协调者询问所有参与者是否可提交
>
> 阶段二（提交 / 回滚）：
>
> 根据投票结果，全部提交或全部回滚
>
> 优缺点：
>
> **✅️ 强一致性**
>
> **❌️ 性能差（同步阻塞）**
>
> ❌️ 协调者单点故障
>
> ❌️ 数据被锁，影响并发
>
> Seata AT 模式（本质不是传统 2PC）：
>
> 基于 2PC 优化，通过 undo log 自动回滚，对业务代码无侵入

**TCC（Try-Confirm-Cancel）**：

> 高并发，需要业务实现回滚逻辑（复杂）
>
> 适用场景：
>
> 高并发、对一致性要求高的场景（如支付、扣库存）
>
> 需要业务实现 3 个接口（阶段）：
>
> Try（预留资源）、Confirm（确认）、Cancel（回滚）
>
> ❗️ 开发成本高，需处理幂等、空回滚、悬挂等问题

**最终一致性（本地消息表 + MQ）**：

> 多数业务场景，允许短暂不一致
>
> 流程：
>
> 1. 业务本地保存消息记录（状态：待发送）
> 2. 业务提交后，发送 MQ 消息
> 3. 消费成功后更新消息状态
> 4. 失败则重试

**MQ 方案的可靠性保证**：

> 生产者：
>
> 事务消息（RocketMQ）/ 本地消息表，保证消息一定发送成功
>
> MQ 本身：
>
> 持久化 + 主从同步，保证消息不丢
>
> 消费者：
>
> 手动 ACK + 幂等处理，保证消息不重复消费

## 5. 分布式 ID 生成器

**要求**：

> 全局唯一、递增（或趋势递增）、高可用、高性能

**常见方案**：

| 方案                  | 原理                     | 优点             | 缺点               |
| --------------------- | ------------------------ | ---------------- | ------------------ |
| 雪花算法（Snowflake） | 时间戳 + 机器 ID +序列号 | 本地生成，性能高 | 依赖时钟           |
| 号段模式（Leaf）      | 批量获取 ID 段           | 简单可靠         | DB 单点            |
| UUID                  | 本地生成                 | 简单             | 无序、长、影响索引 |

**雪花算法（64 bit）**：

| 位     | 用途                        |
| ------ | --------------------------- |
| 1 bit  | 符号位（0）                 |
| 41 bit | 时间戳（毫秒，可用 69 年）  |
| 10 bit | 机器 ID（最多 1024 节点）   |
| 12 bit | 序列号（每毫秒 4096 个 ID） |

## 追问

**CAP 理论的常见误解有哪些？**

> P 不是“是否容忍分区”，而是“分区发生时系统如何表现”，P 不是可选项，网络分区是必然发生的
>
> CAP 讨论的是**当分区发生时**，你在 C 和 A 之间如何选择 

**BASE 和 CAP 的关系？**

> BASE 是对 AP 系统的补充，放弃强一致性，追求最终一致

**RedLock 争议**（Martin Kleppmann VS Redis 作者）：

> 反对：
>
> 依赖系统时钟、安全性存疑、复杂度高
>
> 支持：
>
> 正确使用下足够安全，适合对一致性要求高的场景
>
> 结论：
>
> 大部分业务用 Redission 看门狗 + 单机 Redis 就够了

**本地消息表怎么实现？**

> 本地保存消息记录（状态为“待发送”），业务提交后通过 MQ 发送，消费成功后更新消息状态，失败则重试

## 可选补充

**Paxos VS Raft**：

> 分布式共识算法，Raft 更易懂（选主、日志复制）

| 对比     | Paxos          | Raft                             |
| -------- | -------------- | -------------------------------- |
| 可理解性 | 难（论文难懂） | 易（拆分为选主、日志复制、安全） |
| 工程应用 | 较少           | Etcd、TiKV、Consul               |
| 节点角色 | 无固定角色     | Leader + Follower + Candidate    |

**ZAB 协议**（Zookeeper）：

> 原子广播协议，保证消息顺序
>
> 类似 Raft，但设计更早
>
> 角色：
>
> Leader + Follower

**服务注册与发现**：

> Eureka、Nacos、Consul 区别

| 组件      | 特点                        | CAP     |
| --------- | --------------------------- | ------- |
| Eureka    | 自我保护机制，AP            | AP      |
| Nacos     | 支持 CP / AP 切换，功能丰富 | CP / AP |
| Consul    | 自带健康检查、KV 存储       | CP      |
| Zookeeper | 强一致性，需客户端保持连接  | CP      |

**负载均衡算法**：

> 轮询、随机、一致性哈希（Redis 集群）

| 算法       | 特点                 | 适用场景                  |
| ---------- | -------------------- | ------------------------- |
| 轮询       | 简单，不考虑机器差异 | 机器配置相同              |
| 随机       | 类似轮询             | 同上                      |
| 加权轮询   | 按权重分配           | 机器配置不同              |
| 一致性哈希 | 节点变化时影响小     | 缓存场景（Redis Cluster） |
| 最少连接   | 动态感知机器负载     | 长连接场景                |
