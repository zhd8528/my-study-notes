## 2. 缓存三大问题 + 解决方案

| 问题 | 场景              | 方案                   |区别/关键点|
| ---- | ----------------- | ---------------------- | ---------------------- |
| 穿透 | 查不存在数据      | 布隆过滤器、空值缓存   |拦截不存在请求|
| 击穿 | 热点 key 过期     | 互斥锁、热点永不过期   |单个 key 失效|
| 雪崩 | 大量 key 同时过期 | 过期时间加随机值、集群 |批量 key 失效|

## 3. 持久化

**RDB**：快照，恢复快、文件小，但可能丢数据

**AOF**：追加日志，数据完整，但文件大、恢复慢

> 推荐（4.0+）：
>
> **混合持久化** = RDB 做全量 + AOF 做增量

---

## 4. 淘汰策略

**LRU**

> 最近最少使用

**LFU**

> 最不经常使用

**常用策略**：

- `allkeys-lru`：所有 key 中淘汰最近最少使用（最常用）
- `allkeys-lfu`：所有 key 中淘汰最不经常使用
- `volatile-lru`：只淘汰设置了过期时间的 key

**配置淘汰策略**（redis.conf）：

```bash
maxmemory 4gb
maxmemory-policy allkeys-lru
```

**常见策略选择**：

| 场景                  | 推荐策略                             |
| --------------------- | ------------------------------------ |
| 缓存热点数据          | `allkeys-lru`                        |
| 需要保留最近访问      | `volatile-lru`（只淘汰有过期时间的） |
| 防冷数据污染（4.0 +） | `allkeys-lru`                        |

## 追问

### 1.  缓存三大问题的解决方案细节

**穿透**：查不存在数据

- 布隆过滤器
  - 把所有可能存在的数据 key 放入布隆过滤器，请求先过过滤器，不存在的直接拦截
  - **缺点**：有误判率，可能把存在的判为不存在
- 空值缓存
  - 查询 DB 返回空，也把 null 值缓存（如 value = “”，过期时间短），防止频繁穿透 DB

**击穿**：热点 key 过期

- 互斥锁
  - 使用 SETNX 加锁，只有一个线程去 DB 加载数据，其他线程等待或重试
  - **注意**：要防死锁（加过期时间）
- 逻辑过期
  - 缓存中不设物理过期，而是存一个逻辑过期时间字段
  - 后台线程异步刷新

**雪崩**：大量 key 同时过期

- 过期时间加随机值
  - 比如原过期时间 1 小时，加上0~600秒随机偏移，避免集体失效
- 集群 + 降级限流
  - Redis 集群提高可用性
  - 同时限流防止 DB 被打死

### 2. 持久化细节补充

**RDB如何触发？**

> 手动（save / bgsave）或自己配置，如`save 900 1`

**RDB的缺点？**

> 两次快照间若宕机，会丢失最后一次快照后的数据；
>
> 且快照生成时 fork 子进程，内存占用翻倍（写时复制）

**AOF 的 3 种同步策略？**

> always（每命令同步，最安全但慢）、everysec（每秒同步，推荐）、no（由 OS 决定）

**AOF 重写机制？**

> **解决 AOF 文件膨胀**：bgrewriteaof 将内存中的数据转换为写命令，覆盖旧文件

**混合持久化（4.0+）？**

> RDB 做全量快照 + AOF 记录增量，重启时先加载 RDB 再重放 AOF，兼顾恢复速度和数据完整性

---

### 3. 淘汰策略细节

**Redis 的LRU是严格 LRU 吗？**

> 不是
>
> 它是近似 LRU，随机采样若干 key（默认 5 个），淘汰其中最近最少使用的，避免维护全局链表

**LFU 是什么版本加入的？**

> Redis 4.0，解决 LRU 被冷数据污染问题
>
> LFU 记录访问频率，频率低的先淘汰

**allkeys-lru  VS volatile-lru？**

> allkeys——针对所有 key；
>
> volatile——只针对设置了过期时间的 key

---

### 4. 数据过期策略

**Redis 如何删除过期 key?**

- 惰性删除 + 定期删除
  - **惰性**：访问 key 时检查是否过期，过期则删除
  - **定期**：Redis 每秒执行 10 次（每 100 ms）后台任务，随机抽查一批设置了过期时间的 key，删除其中过期的
    - 不是每 100 ms 都做，而是每 100 ms 执行一次定时任务，该任务会抽查多个 DB

**为什么不用定时器？**

> 如果每个 key 都建定时器，CPU 负载极高

---

### 5. 高可用与容灾

**主从复制如何保证数据最终一致？**

> 全量复制 + 增量复制
>
> 从库通过 replicaof 建立主从，主库发送 RDB + 缓冲区写命令

**哨兵（sentinel）作用？**

> 监控主从、自动故障转移（选举新主）、通知客户端
>
> **主观下线 VS 客观下线**：
>
> - **主观下线**：单个哨兵任务某节点不可达（发送 PING 未回复）
> - **客观下线**：多个哨兵（通常配置 quorum 数量）都认为该节点不可达，触发故障转移

**Cluster 集群？**

> 16384 个哈希槽，去中心化
>
> 数据分片 + 主从切换

---

### 6. 分布式锁

**如何用 Redis 实现分布式锁？**

> SET key value NX EX seconds（原子设置锁 + 过期时间）
>
> 释放时用 Lua 脚本，先 GET 再 DEL，防止误删其他线程的锁

**有什么问题？**

> 锁过期时间不好设置（看门狗方案，如 Redission 自动续期）
>
> Redis 主从异步复制导致锁丢失（需 RedLock 或多节点）

**RedLock的问题**（Martin Kleppmann 质疑）：

> 依赖系统时钟，时钟跳变可能导致锁失效
>
> 性能差，需要多数节点确认
>
> 一般业务用 Redission 的看门狗 + 单机 Redis 就够了

---

### 7. 缓存与数据库一致性

**先更新 DB 还是先删缓存？**

> 先更新 DB，再删缓存
>
> 如果先删缓存，并发下可能有旧数据又写回缓存

**为什么不先删缓存？**

> 1. 线程 A 删缓存 
> 2. 线程 B 读缓存未命中
> 3. 线程 B 读 DB（旧值） 
> 4. 线程 B 写缓存（旧值）
> 5. 线程 A 更新 DB
>
> 结果：
>
> - 缓存一直是旧值
>
> 标准方案：
>
> - 先更新 DB，再删缓存（Cache Aside Pattern）

**删除缓存失败怎么办？**

> 订阅 binlog（Canal） + 消息队列异步重试删除

**为什么不用更新缓存？**

> 更新操作复杂（可能涉及多个 key）。且并发下可能覆盖旧值，删除更简单安全

## 可选补充

### 跳表（Skip List）

> Zset 底层之一，与红黑树对比：
>
> - 实现简单（不需要平衡旋转）
> - 范围查询高效（`ZRANGE`可直接遍历）
> - 平均 O(log n) 复杂度

---

###  Redis 事务

> `MULTI` 开启 / `EXEC` 执行 / `DISCARD` 取消
>
> 不支持回滚（只是命令批量执行）：
>
> - 命令语法错误会全部失败，但执行中错误不影响其他命令

---

### Pipeline

> 批量发送命令，减少 RTT（Round Trip Time）
>
> 注意：
>
> - 非原子性，不支持事务

---

### BigKey 问题

> **发现**：`redis-cli --bigkeys`
>
> **危害**：阻塞 Redis、网络拥塞、集群数据倾斜
>
> **处理**：
>
> - 拆分（如 hash 分片）
> - 替换为更合适的结构（如 string 换 hash）
> - 压缩后存储
